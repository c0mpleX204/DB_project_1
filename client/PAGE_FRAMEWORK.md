# 前端页面设计框架（对应你当前业务）

## 1) 总体目标
先做最小可运行闭环，再逐步扩展真实业务能力：
- 当前阶段：联调 tickets 基础 CRUD，验证前后端通信、错误处理、表单提交流程。
- 下一阶段：切换到真实业务表（flight、ticket_inventory、ticket_order），把页面分成“查询航班、查看库存、提交订单、订单管理”。

## 2) 建议页面结构
1. 首页 Dashboard
- 显示系统状态、今日订单数量、库存预警。
- 快速入口按钮：航班查询、库存管理、下单。

2. 航班查询页 Flights
- 条件：出发地、目的地、日期。
- 结果：航班号、起降时间、价格区间、余票。

3. 库存管理页 Inventory
- 查看每个 ticket_id 的商务舱/经济舱价格与剩余座位。
- 支持管理员更新价格与余票。

4. 订单页 Orders
- 录入 passenger_id、ticket_id、舱位等级，下单。
- 订单列表支持筛选（按乘客、日期、状态）。

5. 旅客页 Passengers
- 基础增删改查，给订单页提供可选乘客。

## 3) 前端代码分层建议
- api 层：只负责请求后端，统一处理 baseURL 和错误。
- page 层：负责页面布局和模块拼装。
- component 层：可复用 UI（表格、表单、状态条）。
- state 层：管理当前页面数据（例如筛选条件、列表、加载中状态）。

## 4) 当前最小页面（本次已实现）
- 机票列表区：GET /api/v1/tickets/
- 新增表单：POST /api/v1/tickets/
- 更新表单：PUT /api/v1/tickets/{id}
- 删除表单：DELETE /api/v1/tickets/{id}
- 日志区：展示每次请求结果，便于排错

## 5) 下一步演进顺序
1. 把 tickets 简化接口替换为真实业务接口（flight/inventory/order）。
2. 把单页面拆分成多页面（Dashboard/Flights/Orders）。
3. 增加登录与角色控制（管理员/普通用户）。
4. 增加表单校验与分页，提升稳定性。
